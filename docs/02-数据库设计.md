# 数据库设计

> D1（Cloudflare SQLite）的表结构设计
> 10 张表，遵循"少表、宽表、必要时 JSON 兜底"原则

---

## ER 概览

```
admins           ──┐
                  │ 1:N
api_keys         ──┤
                  │
                  │
config           ─┘  (单行配置)
                  
departments      ──┐
                  │ 1:N
users (推广人)    ──┤
                  │
                  │ N:M
                  │
tasks (任务)      ──┤
                  │ 1:N
task_targets     ──┤──> users
   │
   │ 1:N
   ↓
click_logs

archived_tasks   ──> tasks（软删后归档）
admin_sessions   ──> admins
```

## DDL

```sql
-- =====================================================
-- 1. 后台管理员
-- =====================================================
CREATE TABLE admins (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  username      TEXT    UNIQUE NOT NULL,
  password_hash TEXT    NOT NULL,           -- PBKDF2 哈希
  created_at    INTEGER NOT NULL,           -- 毫秒时间戳
  last_login_at INTEGER
);

-- =====================================================
-- 2. 系统配置（钉钉参数等，固定单行 id=1）
-- =====================================================
CREATE TABLE config (
  id                       INTEGER PRIMARY KEY DEFAULT 1 CHECK (id = 1),
  dingtalk_corp_id         TEXT,
  dingtalk_app_key         TEXT,
  dingtalk_app_secret      TEXT,
  dingtalk_agent_id        TEXT,             -- 应用 agentid
  dingtalk_access_token    TEXT,             -- 缓存的 corp access_token
  dingtalk_token_expires   INTEGER,          -- token 过期毫秒时间戳
  default_message_type     TEXT DEFAULT 'work_notification', -- 'work_notification' | 'todo'
  updated_at               INTEGER NOT NULL
);

-- =====================================================
-- 3. 钉钉部门（事业部）
-- =====================================================
CREATE TABLE departments (
  dept_id    INTEGER PRIMARY KEY,           -- 钉钉 dept_id
  name       TEXT    NOT NULL,
  parent_id  INTEGER,                       -- 父部门
  path       TEXT,                          -- "公司/销售部/销售一组"
  synced_at  INTEGER NOT NULL
);
CREATE INDEX idx_dept_parent ON departments(parent_id);

-- =====================================================
-- 4. 钉钉员工（推广人）
-- =====================================================
CREATE TABLE users (
  userid     TEXT    PRIMARY KEY,           -- 钉钉 userid
  name       TEXT    NOT NULL,
  mobile     TEXT,
  avatar     TEXT,                          -- 头像 URL
  dept_id    INTEGER,                       -- 主部门
  dept_path  TEXT,                          -- 冗余路径
  title      TEXT,
  is_active  INTEGER DEFAULT 1,             -- 0/1
  synced_at  INTEGER NOT NULL
);
CREATE INDEX idx_user_dept ON users(dept_id);
CREATE INDEX idx_user_active ON users(is_active);

-- =====================================================
-- 5. 推广任务
-- =====================================================
CREATE TABLE tasks (
  id               TEXT    PRIMARY KEY,    -- nanoid(12)
  title            TEXT,
  original_url     TEXT    NOT NULL,
  original_content TEXT,
  receivers_json   TEXT    NOT NULL,        -- {"users":[uid1],"departments":[deptId1]}
  message_type     TEXT    NOT NULL,        -- 'work_notification' | 'todo'
  status           TEXT    NOT NULL DEFAULT 'draft',  -- draft|published|archived
  polished_json    TEXT,                    -- {"friend_circle":"...","group":"...","private":"..."}
  created_by       TEXT    NOT NULL,        -- 'agent' | admin username
  creator_id       TEXT,                    -- admin id 或 'agent'
  source           TEXT,                    -- 'agent' | 'dashboard'
  created_at       INTEGER NOT NULL,
  updated_at       INTEGER NOT NULL,
  published_at     INTEGER,
  deleted_at       INTEGER                  -- 软删时间
);
CREATE INDEX idx_task_status     ON tasks(status);
CREATE INDEX idx_task_created_at ON tasks(created_at);
CREATE INDEX idx_task_deleted_at ON tasks(deleted_at);

-- =====================================================
-- 6. 任务-推广人 中间表（每个推广人一个短链/二维码/文案）
-- =====================================================
CREATE TABLE task_targets (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id           TEXT    NOT NULL,
  userid            TEXT    NOT NULL,
  short_code        TEXT    UNIQUE,
  short_url         TEXT,
  qr_r2_key         TEXT,
  copy_used         TEXT,
  copy_type         TEXT,                   -- 'friend_circle' | 'group' | 'private'
  dingtalk_msg_id   TEXT,
  sent_at           INTEGER,
  send_status       TEXT,                   -- 'pending' | 'success' | 'failed'
  send_error        TEXT,
  UNIQUE(task_id, userid)
);
CREATE INDEX idx_target_task ON task_targets(task_id);
CREATE INDEX idx_target_user ON task_targets(userid);
CREATE INDEX idx_target_code ON task_targets(short_code);

-- =====================================================
-- 7. 点击日志
-- =====================================================
CREATE TABLE click_logs (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  short_code   TEXT    NOT NULL,
  task_id      TEXT    NOT NULL,
  userid       TEXT    NOT NULL,
  clicked_at   INTEGER NOT NULL,
  ip           TEXT,
  user_agent   TEXT,
  browser      TEXT,
  os           TEXT,
  device_type  TEXT,                        -- 'mobile'|'pc'|'tablet'
  referer      TEXT,
  country      TEXT,
  city         TEXT
);
CREATE INDEX idx_click_task ON click_logs(task_id);
CREATE INDEX idx_click_user ON click_logs(userid);
CREATE INDEX idx_click_code ON click_logs(short_code);
CREATE INDEX idx_click_time ON click_logs(clicked_at);

-- =====================================================
-- 8. 归档任务
-- =====================================================
CREATE TABLE archived_tasks (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id      TEXT    NOT NULL,
  snapshot     TEXT    NOT NULL,            -- 任务 + targets + stats 的 JSON 快照
  archived_at  INTEGER NOT NULL,
  archived_by  TEXT
);
CREATE INDEX idx_arch_task ON archived_tasks(task_id);

-- =====================================================
-- 9. API Key
-- =====================================================
CREATE TABLE api_keys (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  name          TEXT    NOT NULL,
  key_hash      TEXT    NOT NULL,           -- SHA-256 哈希
  key_prefix    TEXT    NOT NULL,           -- 前 8 位
  created_at    INTEGER NOT NULL,
  last_used_at  INTEGER,
  is_active     INTEGER DEFAULT 1
);

-- =====================================================
-- 10. 管理员 Session
-- =====================================================
CREATE TABLE admin_sessions (
  token       TEXT    PRIMARY KEY,
  admin_id    INTEGER NOT NULL,
  created_at  INTEGER NOT NULL,
  expires_at  INTEGER NOT NULL,
  ip          TEXT
);
CREATE INDEX idx_sess_admin ON admin_sessions(admin_id);
```

## 关键设计决策

### 1. 任务数据 = tasks + task_targets 两表
- `tasks` 存任务元数据
- `task_targets` 一对多，存每个推广人的专属短链/二维码/文案/推送状态
- 避免在 tasks 里塞 JSON 数组（SQLite JSON 数组操作性能差）

### 2. JSON 字段用 TEXT 存
- `receivers_json`、`polished_json`：避免多表 join
- SQLite 对 JSON 函数有 `json_extract`，够用

### 3. click_logs 冗余 task_id 和 userid
- 统计时不用 join task_targets
- 提升查询性能（统计是高频操作）

### 4. 时间统一用毫秒时间戳 INTEGER
- D1/SQLite 没有原生 datetime
- 毫秒时间戳避免精度问题（CF Workers `Date.now()` 是毫秒）

### 5. 软删用 deleted_at 字段
- 回收站列表：`WHERE deleted_at IS NOT NULL`
- 正常列表：`WHERE deleted_at IS NULL`

### 6. 归档独立表
- 归档是 JSON 快照，不再拆多表
- 简化彻底删除逻辑（DELETE FROM archived_tasks WHERE id = ?）

### 7. 密码哈希
- Web Crypto API PBKDF2-SHA256，10 万次迭代
- 零依赖，纯 Workers 内置

### 8. 索引策略
- 高频查询字段加索引
- 复合索引按需（先用单列索引）

## D1 的限制（开发注意）

| 限制 | 影响 |
|---|---|
| 单数据库最大 10GB（付费版）/ 500MB（免费版） | 小项目够用 |
| 单次写入最多 100MB | 大批量插入要分批 |
| 写入有 QPS 限制 | 同步用临时表 + 原子切换避免锁 |
| 字符串不区分大小写（默认） | username 等需要 `COLLATE BINARY` 区分大小写时显式声明 |
| 无布尔类型 | 用 INTEGER 0/1 |

## 初始化脚本

首次部署时执行 `wrangler d1 execute <DB_NAME> --file=src/db/schema.sql`

后续 schema 变更用 `migrations/` 目录管理：
```
migrations/
├── 0001_initial.sql
├── 0002_add_xxx.sql
└── ...
```

每次变更记录到 `项目进度.md` 决策记录。
