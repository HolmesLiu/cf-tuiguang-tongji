# 业务模型

> 本文档描述 cf推广统计 平台的完整业务模型

---

## 1. 角色 & 权限

| 角色 | 是否登录平台 | 干啥 |
|---|---|---|
| **发布人 / 操作人** | ✅ | 通过 agent 对话或后台直接发布任务，查看自己任务效果 |
| **管理员（使用人）** | ✅ | 后台看**所有任务、所有推广人**数据、删除任务、配置钉钉参数、管理 API Key |
| **推广人** | ❌ 不登录 | 只通过**钉钉**收到推广消息 + 二维码/短链 |
| **Agent（娜娜子）** | ❌ 走 API Key | 调平台 API 发任务、查数据 |

> **说明**：发布人和管理员是后台用户的两种角色，权限不同：
> - 发布人：只能看自己的任务、只能发任务
> - 管理员：可看所有任务、可删任务、可配置系统
> MVP 阶段简化为统一管理员，后续按需拆分。

## 2. 任务接收方规则

任务发布人下达的接收方有三种组合：

| 发布人下达 | 实际接收 |
|---|---|
| `推广人: 张三` | → 张三 一人 |
| `推广事业部: 销售一部` | → 销售一部下**所有员工**（按部门树展开） |
| `推广人: 张三, 推广事业部: 销售一部` | → 张三 ∪ 销售一部全部（**去重**） |
| （后台手动多选） | → 后台多选的人员 |

实现逻辑：
- 系统拉取钉钉部门 + 员工的全量快照到本地 D1
- 发布任务时，按 `receivers_json`（含 userid 列表 + 部门 id 列表）展开成最终推广人列表
- 同一人只生成一条短链（去重）

## 3. 任务核心实体

```
Task（任务）
├── id (nanoid 12)
├── title              任务标题
├── original_url       原始链接（要推广的 URL）
├── original_content   原内容（agent 润色前的）
├── receivers_json     接收方规则 {"users":[uid1],"departments":[deptId1]}
├── message_type       'work_notification' | 'todo'
├── status             'draft' | 'published' | 'archived'
├── polished_json      润色后的文案 {"friend_circle":"...","group":"...","private":"..."}
├── created_by         'agent' | admin username
├── source             'agent' | 'dashboard'
├── created_at / updated_at / published_at
└── deleted_at         软删时间（NULL=正常）

TaskTarget（任务-推广人 中间表，每个推广人一行）
├── task_id
├── userid             推广人钉钉 userid
├── short_code         8 位 base62
├── short_url          完整短链
├── qr_r2_key          R2 路径，如 "qr/abc123.png"
├── copy_used          实际使用的文案
├── copy_type          'friend_circle' | 'group' | 'private'
├── dingtalk_msg_id    钉钉消息 id
├── sent_at
├── send_status        'pending' | 'success' | 'failed'
└── send_error
```

## 4. 业务流程

### 4.1 任务发布主流程

```
操作人(agent 聊天) ──> 任务参数
    ↓
Agent 解析 + 润色文案（朋友圈/群发/私聊 三版）
    ↓
POST /api/agent/tasks  ──> 平台创建任务(status=draft, polished_json 已填充)
    ↓
Agent / 操作人 确认  ──>  POST /api/agent/tasks/:id/publish
    ↓
平台发布流程：
  1. 接收方规则展开（用户 + 部门 → 最终 userid 列表，去重）
  2. 对每个推广人：
     a. 生成 short_code（8位 base62）
     b. 拼出 short_url = https://tuiguang.7kk.top/s/{code}
     c. 用 qrcode 包生成 PNG，上传 R2
     d. 选定该推广人的 copy_type + copy_used
  3. 写入 task_targets 表（每个推广人一行）
  4. 调钉钉 API：
     - work_notification → topapi/message/corpconversation/asyncsend_v2
     - todo             → topapi/workrecord/add
     推送内容含：短链 + 二维码图片（来自我们 R2）
  5. 更新 send_status
  6. 任务 status → 'published'
    ↓
返回结果给 agent / 后台
```

### 4.2 短链点击 & 监控

```
推广人微信里点开链接（点短链 / 扫二维码 → 短链）
    ↓
GET /s/{code}  ──>  Worker 短链中转
    ↓
解析：
  1. 从 KV/D1 查 short_code → task_id, userid
  2. 拿 cf-connecting-ip / user-agent / referer
  3. cf-ipcountry / cf-ipcity
  4. ua-parser-js 解析 → browser, os, device_type
  5. 写 click_logs
    ↓
302 跳转到 original_url
```

### 4.3 后台统计

```
管理员登录后台
    ↓
GET /api/tasks/:id/stats
    ↓
聚合 click_logs：
  - 总点击数（count）
  - 独立 IP 数（count distinct ip）
  - 独立推广人数（count distinct userid）— 应该等于 task_targets 数
  - 按 userid 聚合 → 每个推广人的点击/IP
  - 按 device_type / country / os 聚合
  - 按小时/天 聚合 → 趋势
    ↓
返回 JSON + 图表渲染
```

## 5. 钉钉集成

### 5.1 鉴权

- **企业 access_token**：从 appkey + appsecret 换
  - URL：`https://oapi.dingtalk.com/gettoken?appkey=***&appsecret=***`
  - 有效期：7200 秒
  - 缓存策略：存 config 表 + expires_at 字段，**请求时检查，过期前 5 分钟主动刷新**
  - 同时也存 KV（加速读）

### 5.2 通讯录同步（分批 + 全量）

**全量同步**（每次都全量重建，保证无重复）：

```
POST /api/contacts/sync 触发
    ↓
1. 清空临时表 users_tmp / departments_tmp（先同步到临时表）
2. 分批拉取部门列表：
   - GET /topapi/v2/department/listsub?dept_id=1&cursor=...&size=100
   - 递归：每个部门再 listsub
   - 全部存 departments_tmp
3. 分批拉取用户列表：
   - GET /topapi/v2/user/listsub?dept_id=***&cursor=...&size=100
   - 每个部门都拉
   - 全部存 users_tmp
4. 原子切换：RENAME users_tmp → users, departments_tmp → departments
5. 记录 synced_at
6. 清理 userid 不在新数据中的推广人相关的 task_targets？不！保留历史任务数据
```

**失败处理**：
- 每批失败重试 3 次
- 整个同步失败 → 临时表丢弃，保留旧表
- 同步期间后台可读旧数据（不阻塞）

### 5.3 推送

| 消息类型 | 接口 | 鉴权 |
|---|---|---|
| 工作通知 | `POST /topapi/message/corpconversation/asyncsend_v2?access_token=***` | corp access_token |
| 待办 | `POST /topapi/workrecord/add?access_token=***` | corp access_token |
| 通讯录 | `POST /topapi/v2/department/listsub` 等 | corp access_token |

**确认结论（2026-06-08 核对）**：
- "工作通知" 和 "待办" 都**不需要用户 OAuth 授权**
- 之前印象中"发待办需要用户授权"是混淆了第三方个人应用场景
- **整段 OAuth 流程砍掉**

### 5.4 消息内容

钉钉工作通知采用 `markdown` 类型或 `action_card`：

```markdown
### 📣 推广任务：{title}

**原始链接**：{original_url}

**专属推广链接**：{short_url}
**专属二维码**：（来自 R2 的图片）

---

**朋友圈版文案**：
{copy_friend_circle}

**群发版文案**：
{copy_group}

**私聊版文案**：
{copy_private}
```

## 6. 监控维度

### 6.1 任务维度（每个任务）

- 总点击数 / 独立 IP / 独立推广人数
- 时间趋势（小时 / 天）
- 设备分布（手机/PC/平板）
- 地域分布（国家 / 城市）
- 浏览器 / OS 分布
- 推广人排行（按独立 IP 排序）

### 6.2 推广人维度（每个推广人在每个任务下）

- 该推广人带来的：总点击 / 独立 IP
- **推广详单**（每一条点击记录）：
  - 时间
  - IP
  - 浏览器 / OS / 设备
  - **来源地址（referer）** ← 大宇明确指定这个作为"渠道"
  - 推广任务 / 推广人

## 7. 任务删除（软删 + 归档 + 彻底删）

```
任务状态机：
draft ──发布──> published ──软删──> archived ──彻底删──> (消失)
                      │                                        ↑
                      └──────────还原──────────────archived ──┘
```

- **软删**：`tasks.deleted_at = now()`，从主列表移除，进归档
- **归档**：`tasks` 数据复制到 `archived_tasks` 表（JSON 快照）
- **还原**：`archived_tasks` 恢复回 `tasks`
- **彻底删**：`archived_tasks` 物理删除

## 8. 二维码复制问题

**问题**：钉钉里复制二维码图片到微信不能直接用
**原因**：钉钉 CDN 图片有防盗链，跨 IM 平台粘贴失效
**方案**：图片源**必须来自我们自己域名**（`tuiguang.7kk.top/qr/xxx.png`），钉钉消息里以 markdown 内嵌图片形式提供

**用户体验**：
1. 钉钉消息直接看得到二维码（预览）
2. 推广人长按图片 → 保存到相册 → 微信"相册"发图（最快）
3. 或点消息里的短链 → H5 详情页 → 长按保存 / 复制链接
