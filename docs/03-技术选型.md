# 技术选型

> 选型原则：**最低成本、最高效率、纯 Cloudflare 生态**

---

## 总览

| 类别 | 选型 | 备选 | 理由 |
|---|---|---|---|
| 运行时 | Cloudflare Workers (TS) | — | 唯一选择 |
| 数据库 | Cloudflare D1 (SQLite) | KV | 关系型 + 聚合统计 |
| 对象存储 | Cloudflare R2 | KV | 二维码图片 |
| 缓存 | Cloudflare KV | D1 | access_token 高频读 |
| 静态资源 | Workers Static Assets | Pages | 一体化部署 |
| HTTP 客户端 | Workers 原生 fetch | axios | 零依赖 |
| 钉钉 SDK | 原生 fetch | dingtalk-sdk | 第三方包体积大且未必兼容 Workers |
| 二维码 | `qrcode` | qr-image | 成熟稳定 |
| UA 解析 | `ua-parser-js` | ua-parser | 浏览器/OS/设备 |
| 短码 | `nanoid` | 自实现 | 8位 base62 够用 |
| 密码哈希 | Web Crypto PBKDF2 | bcryptjs | 零依赖，Workers 原生 |
| 参数校验 | `zod` | joi | TS 友好，体积小 |
| 前端框架 | 原生 JS | Alpine/htmx | 后台简单，原生够用 |

## 详细说明

### 1. Cloudflare Workers
- **版本兼容性**：`nodejs_compat` 标志（启用部分 Node API）
- **CPU 时间限制**：免费版 10ms，付费版 50ms
- **冷启动**：<5ms
- **语言**：TypeScript
- **部署工具**：`wrangler`

### 2. Cloudflare D1（SQLite）
- **优势**：SQL 关系型，聚合查询快
- **限制**：单 DB 10GB、500 万次读/天（免费版）
- **接口**：从 Worker 通过 binding 访问，类型安全

### 3. Cloudflare R2
- **优势**：S3 兼容 API，无出网费
- **用途**：存二维码图片
- **公开访问**：通过 Worker 代理（不直接公开 R2）

### 4. Cloudflare KV
- **优势**：全球分布，最终一致性
- **用途**：缓存 corp access_token、短码查 target
- **不存**敏感数据

### 5. Workers Static Assets
- **优势**：wrangler 一个项目里既跑代码又托管静态资源
- **替代**：CF Pages（但要分两个项目部署）
- **配置**：`wrangler.toml` 加 `assets` 段

### 6. 钉钉 API
- **走原生 fetch**，不依赖第三方 npm 包
- **理由**：钉钉 SDK 体积大、依赖 Node API、未必兼容 Workers
- **代价**：需要自己封几个方法（已经在 `dingtalk/client.ts` 里规划）

### 7. npm 包
- `qrcode`：~30kb minified，CF Workers 兼容
- `ua-parser-js`：~20kb，CF Workers 兼容
- `nanoid`：~1kb，CF Workers 兼容
- `zod`：~12kb，CF Workers 兼容

所有包都需要在 `package.json` 里，并经过 wrangler 打包。**避免任何带 Node 内置依赖的包**（如 `fs`、`crypto`）。

### 8. 前端（管理后台）
- **方案 A（推荐）**：原生 JS + 模块化
  - 一个 `app.html` + 一堆 JS 模块
  - 模板字符串拼界面
  - 适合 CRUD 风格的后台
- **方案 B**：Alpine.js / htmx
  - 极轻量（~15kb）
  - 渐进增强
- **方案 C**（不推荐）：React/Vue
  - 需要 build 工具，部署复杂
  - 大材小用

**MVP 选 A**，后续视情况加 B。

## 不选的项目 & 理由

| 不选 | 理由 |
|---|---|
| 第三方服务器 | 大宇明确要求"小项目，最低成本，不上其他服务器" |
| Docker | CF Worker 不需要容器化 |
| Redis | KV 够用 |
| MySQL/PostgreSQL | D1 够用 |
| 第三方短链服务 | 自己实现，成本为 0 |
| 第三方二维码服务 | 自己生成，零成本 |
| 钉钉 SDK | 体积 + 兼容性问题 |

## 成本估算（小项目）

| 服务 | 免费额度 | 预期用量 | 月成本 |
|---|---|---|---|
| Workers | 10 万次/天 | ~1-2 万次/天 | **$0** |
| D1 | 500 万读/天 + 5GB | 几十万读/天 + <100MB | **$0** |
| R2 | 10GB 存储 + 1000 万读/月 | <1GB + <10 万读 | **$0** |
| KV | 10 万读/天 | 几千读/天 | **$0** |
| 域名 | — | 已有 7kk.top | **$0** |
| **合计** | | | **$0/月** |

付费起算点（仅在规模增长后）：
- Workers：1000 万次/月 后 $0.50/百万
- D1：超出免费额度后 $0.75/GB 存储 + $1/百万读
- R2：超出后 $0.015/GB 存储

## Workers 兼容包清单

部署前要验证每个 npm 包都兼容 Workers：

| 包 | 兼容性 | 备注 |
|---|---|---|
| `qrcode` | ✅ | 纯 JS |
| `ua-parser-js` | ✅ | 纯 JS |
| `nanoid` | ✅ | 纯 JS |
| `zod` | ✅ | 纯 JS |
| `bcryptjs` | ⚠️ 需测试 | 有 `crypto` 依赖，但 Workers 有 nodejs_compat |
| `axios` | ❌ 不需要 | 用 fetch |

**降级方案**：所有依赖如出问题，回退到 Web Crypto API + 原生 fetch。
